# russia-flights — контекст проекта

## Что это
Интерактивная карта (Leaflet) прямых международных рейсов из городов России.
Данные обновляются автоматически через GitHub Actions, без ручного вмешательства в штатном режиме.

Репозиторий: github.com/xkonstantinnnx/russia-flights

## Архитектура данных
Конвейер из источников. Правило: каждый следующий источник ТОЛЬКО ДОБАВЛЯЕТ данные, никогда не удаляет.

1. **AirLabs** (`run_airlabs_primary`) — основной источник, все аэропорты, постраничная загрузка `limit=50&offset=N`.
2. **AeroDataBox** (через RapidAPI) — дополняющий источник, фильтр `countryCode != RU`.
3. **OpenSky** (`run_opensky_additive`) — дополняющее обогащение, только добавляет записи.
4. **Яндекс.Расписания** — третий дополняющий источник (введён после отключения gogov.ru).

Обработка в два этапа: **Stage1 → Stage2**, с ранним выходом (early exit) на Stage1.
Перед запуском текущий `routes.json` загружается с диска; маршруты, не найденные ни одним источником, удаляются — это единственное место, где данные пропадают.

## Деплой
- Прод: https://russia-flights.ru/ — виртуальный (shared) хостинг Timeweb.
- GitHub Actions (`update.yml`) только обновляет `routes.json` и пушит в репозиторий. Прямой FTP-деплой из этого workflow убран 2026-04-15 после серии падений `Timeout (control socket)` на всех протоколах (FTP/FTPS/SFTP) — похоже, файрвол Timeweb блокирует AWS IP-диапазон раннеров целиком, не только SSH.
- Единственный реальный механизм публикации на прод — Crontab самого виртуального хостинга russia-flights.ru: он сам забирает свежий `routes.json` из GitHub. Это не резервный, а ОСНОВНОЙ путь деплоя.
- SSH-доступ к виртуальному хостингу: `ssh -i ~/.ssh/timeweb_shared [REDACTED-USER]@[REDACTED-HOST]`. Сайт живёт в `~/russia-flights/public_html/`. Crontab хостинга можно проверить через `crontab -l` по этому SSH.
- Диагностические workflow `Test SSH` и `Тест всех вариантов деплоя Timeweb` (`test_proxy.yml`) — ручные (`workflow_dispatch`), для проверки протоколов. На 2026-04-12 все методы деплоя (FTP plain/FTPS×2/SFTP/rsync) падали. Не предлагать SSH-деплой из CI.

## Жёсткие правила для index.html
- `loadRoutes()` обязан проверять пустой `routes.json` и откатываться на `SEED`.
- `const RU_COORDS` должен быть объявлен ДО блока `/* ── MARKERS ── */`.
- `function getRUCoords` идёт сразу после `RU_COORDS`.
- Не менять порядок этих блоков даже при рефакторинге — от этого зависит инициализация карты.

## Инфраструктура для отладки (только личная, не рабочая)
- Логи пайплайна — через GitHub Actions: `gh run list`, `gh run view <id> --log`.
- Личный VPS, алиас `timeweb-flights-vps` в `~/.ssh/config` — НЕ имеет отношения к деплою russia-flights. Это отдельный сервер с другими личными проектами (russiametro, vikunja, openproject, zabbix); cron/скриптов для этого проекта там нет.
- Cron, реально публикующий прод, живёт в Crontab виртуального хостинга russia-flights.ru — туда нет SSH-доступа (см. «Деплой»).
- В этом проекте нет и не будет доступа к рабочей/корпоративной инфраструктуре — только этот VPS и GitHub.

## Чего агенту нельзя делать без явного подтверждения
- `git push` в любую ветку.
- Запуск workflow вручную (`gh workflow run`).
- Любые изменения FTP-credentials и секретов в GitHub Actions — это всегда руками, не агентом.
- Удаление маршрутов из `routes.json` напрямую (только через штатный пайплайн).
